The problem is the interaction between Hypothesis's shrinking (which pushes values toward 0) and C++'s
std::normal_distribution implementation.

C++ std::normal_distribution (in libstdc++) uses the Marsaglia polar
method, which is rejection-based:

1. Generate two uniform values u1, u2 in (-1, 1)
2. Compute s = u1² + u2²
3. If s >= 1 or s == 0, reject and retry

When Hypothesis checks the "smallest natural input," it tries all-zero
engine outputs. With all zeros, generate_canonical produces 0.0, so u1 =
u2 = 0.0, s = 0, and the rejection condition hits every time. The
distribution loops forever consuming draws, and Hypothesis sees a test
that needs a huge amount of data.

Rust's rand crate uses the ziggurat method for StandardNormal, which
doesn't have this degenerate case with zero inputs — it terminates
quickly regardless of the input value.

This means artificial mode is fundamentally incompatible with C++
rejection-based <random> distributions. The practical fix is to default
use_true_random to true, since that's the only mode that works reliably
with <random> distributions. Users who want artificial mode would use it
for direct operator() calls, not through distributions.

Safe (no rejection, no infinite loop on all-zero input):
- uniform_real_distribution — direct scaling 
- exponential_distribution — inverse transform: -log(1-U)/λ
- weibull_distribution — inverse transform 
- cauchy_distribution — inverse transform via tan()                      
- extreme_value_distribution — inverse transform                         
- bernoulli_distribution — threshold comparison                          
- geometric_distribution — inversion                                     
                                                                        
Infinite loop on all-zero input (rejection-based or inherits from one):  
- normal_distribution — Marsaglia polar method (rejects when s == 0)     
- lognormal_distribution — wraps normal_distribution                     
- gamma_distribution — acceptance-rejection (for shape ≥ 1)
- chi_squared_distribution — wraps gamma_distribution
- student_t_distribution — wraps normal_distribution + chi_squared_distribution
- fisher_f_distribution — wraps chi_squared_distribution
- poisson_distribution — rejection for mean ≥ 12
- binomial_distribution — rejection for large t×p

Design

 Two sides to this feature:

 1. Producing the blob: After finding a failure, the server encodes the shrunk choices and includes the blob in the test_done results.
 2. Consuming the blob: The SDK passes a blob in run_test, the server skips ConjectureRunner and directly replays the pre-seeded choices.

 Uses Hypothesis internals encode_failure/decode_failure from hypothesis.core (base64 + zlib + custom binary serialization of the choice sequence). No version pinning for now.
